# SomMark Plugin System

SomMark is built with a modular, plugin-driven architecture. Almost every core feature—from module imports to comment stripping—is implemented as a plugin. This guide explains how to create and use plugins to extend SomMark's transpilation lifecycle.

## 1. Plugin Structure

A plugin is a JavaScript object that defines its capabilities and hooks into specific stages of the transpilation process.

```javascript
const myPlugin = {
  name: "my-custom-plugin",
  type: "transform", // can be a string or an array of strings
  author: "Your Name",
  description: "A brief description of what the plugin does.",
  
  // Hooks (depending on type)
  transform(output) {
    return output.toUpperCase();
  }
};
```

### Core Properties
- `name` (required): A unique identifier for the plugin.
- `type` (required): The lifecycle stage(s) this plugin hooks into.
- `options`: Default configuration for the plugin.

## 2. Lifecycle Hooks

SomMark provides hooks at every major stage: Source → Code → Tokens → AST → Target String.

### Preprocessor (`preprocessor`)
Modifies the source text before the lexer sees it. Can be scoped to specific regions.

* **Hook**: `beforeLex(src)`
* **Scopes**:
  - `top-level`: The entire raw source.
  - `arguments`: The content inside block/inline arguments (e.g., `[Tag=... arguments ...]`).
  - `content`: The text body between block tags.

```javascript
const shoutPreprocessor = {
  name: "shout",
  type: "preprocessor",
  scope: "content",
  beforeLex(src) {
    return src.toUpperCase();
  }
};
```

### After Lexer (`lexer`)
Manipulates the token stream after the lexer has run.

* **Hook**: `afterLex(tokens)`
* **Input**: Array of token objects `{ type, value, metadata }`.

### On AST (`parser`)
Modifies the Abstract Syntax Tree (AST) before it is sent to the transpiler.

* **Hook**: `onAst(ast, { mapperFile })`
* **Use Case**: Validation (like `rules-validation`) or structural optimization.

### Mapper Extensions (`mapper`)
Statically adds new tags or global rules to the active mapper.

* **Hook**: `outputs` (Array) / `rules` (Object)
* **Description**: Allows a plugin to "own" certain tags.

```javascript
const tablePlugin = {
  name: "data-table",
  type: "mapper",
  outputs: [
    { id: "Grid", render: ({ content }) => `<table>${content}</table>` }
  ],
  rules: {
    recognizedArguments: ["border", "spacing"]
  }
};
```

### Transformer (`transform`)
Modifies the final string output after transpilation is complete.

* **Hook**: `transform(output)`
* **Use Case**: Minification, syntax highlighting, or post-processing.

### Register Hook (`register`)
Provides programmatic access to the `SomMark` instance before processing starts.

* **Hook**: `register(sm)` or `registerOutput(sm)`

## 3. Plugin Priority

By default, built-in plugins run first. You can control the exact execution order using the `priority` array when calling `transpile`.

```javascript
import { transpile } from "sommark";

const output = await transpile({
  src: "...",
  plugins: [pluginA, pluginB],
  priority: [pluginB, pluginA] // pluginB will run before pluginA
});
```

## 4. Built-in Plugins

- **`module-system`**: (Active by default) Handles `[[import]]` and `$[[key]]`.
- **`comment-remover`**: (Active by default) Strips `# comments`.
- **`quote-escaper`**: (Active by default) Handles auto-escaping in quoted strings.
- **`rules-validation`**: Validates tags against rules defined in mappers.
- **`raw-content`**: Treats specific blocks as literal text (e.g., `<script>`, `<style>`).
- **`sommark-format`**: Adds support for `.smark` specific formatting rules.
